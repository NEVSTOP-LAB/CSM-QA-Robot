# 代码评审（codex · 2026-04-10）

## 结论
- 主流程模块化清晰，日志与告警覆盖核心路径，但部分关键场景实现与预期不符（尤其是 question 类型处理与文章摘要），会直接导致监控目标缺失或检索效果偏离。

## 主要发现
1) **question 类型无法正常读取/发布**  
   - `scripts/zhihu_client.py:148-168` 在 object_type=`question` 时使用 `/answers/{id}/comments`，但 `config/articles.yaml` 提供的是 question id 而非 answer id，且 `_expand_articles()` 未将 question 展开为具体回答（`scripts/run_bot.py:639-683`），导致 question 类型目标会请求错误的 URL。  
   - 自动发布也将 question 映射为 `object_type="answer"` 但仍传入 question id（`scripts/run_bot.py:504-516`），同样不符合知乎写评论接口对 answer_id 的要求。结果是 question 场景下既无法抓取评论，也无法发布回复，应在 question 时先获取回答 ID 并用正确的 answer_id 调用读/写接口。

2) **文章摘要生成未使用正文，效果形同缺失**  
   - `process_article()` 调用 `summarize_article()` 时将 `title` 同时作为标题和正文传入（`scripts/run_bot.py:300-308`），未提供真实文章内容，生成的“摘要”只能是标题的复述，无法为线程元数据或 RAG 提供背景。与计划中“生成简短摘要便于上下文理解”不符，需要提供文章正文或至少简介作为摘要输入。

3) **真人回复索引的 question 取值可能错位**  
   - 在捕获作者真人回复时，仅取最近两轮上下文并使用 `messages[0]` 作为 question（`scripts/run_bot.py:581-589`）。当最近两轮是“Bot 回复 → 真人回应”时，`messages[0]` 会是上一轮 Bot 内容，导致索引记录的 QA 对中 question 部分被 Bot 回复占据，后续检索会召回与真实用户提问无关的文本。建议在真人回复分支明确选择最近一条用户内容作为 question。
